# UI design spec implementation

## 背景

ユーザーから、提示画像をもとに整理された「ビリビリレーサーズ デザイン仕様書」を受け取り、直前の `/plan` で実装計画を提示した。その後 `go` 指示があったため、既存の静的 Web アプリに仕様を反映する。

## 目的

`app/web` の一人プレイ初版を、仕様書の主要画面、共通 UI、courseId ベースのマスタ管理、PC/タッチ HUD の考え方に沿った実装へ拡張する。

## タスク種別

機能追加

## スコープ

- `app/web/index.html`、`app/web/app.js`、`app/web/styles.css` の UI/画面状態/ゲーム表示を更新する。
- `app/web/README.md` に実装済み画面、未実装オンライン機能、起動・検証方法を反映する。
- 完了レポートを `reports/working/` に保存する。

## スコープ外

- 実際のオンライン対戦、外部 API、永続化、ランキング共有の実装。
- 画像アセットや 9-slice 画像素材の新規制作。
- 完全な衝突判定マスク/タイルマップ化。

## 実装計画

1. 既存画面構成を、仕様の主要フローに合わせて `HOME`、`TEAM_CREATE`、`JOIN_BY_CODE`、`MAP_LIST`、`RACE_LOBBY`、`GAMEPLAY`、`RESULT` 相当に整理する。
2. `CourseMaster` 相当の courseId ベースデータを導入し、表示名・説明・難易度・ラップ・記録・テーマをマスタ参照にする。
3. 未実装オンライン導線は、実データのような固定値ではなく、明示的な「準備中」「未接続」「ローカルプレビュー」状態として表示する。
4. PC HUD とタッチ HUD の表示を同一レース状態から切り替えられる形へ整える。
5. README と作業レポートを更新し、最小十分な検証を実行する。

## ドキュメント保守計画

- UI と操作方法が変わるため `app/web/README.md` を更新する。
- `docs/` は存在しないため、永続ドキュメント更新は README に限定する。

## 受け入れ条件

- [ ] HOME から主要画面へ遷移でき、仕様書の主要文言・導線が反映されている。
- [ ] コース表示は `courseId` を正とするマスタから描画され、表示名の直書き分散が減っている。
- [ ] TEAM_CREATE、JOIN_BY_CODE、MAP_LIST、RACE_LOBBY、GAMEPLAY、RESULT の主要情報が仕様に沿って表示される。
- [ ] オンライン未実装の参加、フレンド、ランキング共有などを実データのように見せず、未実装またはローカルプレビューとして明示している。
- [ ] PC/タッチ向け HUD の表示方針が実装に反映されている。
- [ ] README が新しい画面・制約・起動方法と整合している。
- [ ] 変更範囲に対して `git diff --check` と静的 Web の起動確認を実行し、結果を記録している。

## 検証計画

- `git diff --check`
- `python3 -m http.server 4173 --directory app/web` を起動し、`curl -I http://127.0.0.1:4173/` で静的配信を確認する。
- 可能であれば headless Chrome で主要画面のロードを確認する。

## PR レビュー観点

- 仕様を満たすための固定サンプル表示が、本番実データに見える UI になっていないこと。
- コース名・記録・プレイヤー情報のデータ由来が明確であること。
- 画面遷移とキーボード操作が既存一人プレイを壊していないこと。
- README の未実装範囲が実装と一致していること。

## リスク

- 静的アプリのため、オンライン機能は UI 状態の表現に留まる。
- 画像アセットがないため、コースプレビューは CSS/canvas 表現で代替する。
- 仕様書すべての詳細画面を完全実装すると変更量が大きいため、主要画面とデータ構造を優先する。
